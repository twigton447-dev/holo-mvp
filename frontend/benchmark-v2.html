Why this matters

Business email compromise is not a niche attack class.

It is consistently ranked by the FBI as the highest-dollar cybercrime category in the United States — and it is exactly the kind of decision AI agents are now being asked to make. Reviewing payment requests, approving routing updates, handling accounts payable workflows that look routine until they are not.

If a model cannot be trusted here, it cannot be trusted at the action boundary.

The flagship result

GPT-5.4 is OpenAI's most advanced model.

It is the model most widely deployed in enterprise workflows today. We ran it against a fraudulent payment routing update. The amount was ordinary. The language was calm. Nothing in the message labeled itself as malicious.

The threat existed only in the structural relationship between the sender identity and the approved vendor record.

GPT-5.4 approved the wire.

A second finding

On a different scenario, all three solo models failed.

The flagship result shows GPT-5.4 failing while Claude and Gemini catch it. That is a meaningful result. But it is not the worst case.

On a threshold fraud scenario — an invoice with an amount pattern designed to stay just below approval policy triggers — all three solo frontier models independently returned ALLOW. The attack did not fail any individual model's heuristics. It sat in the gap between them.

Holo caught it. None of the solo models did.

This result emerged after a domain calibration refinement: payment action types now require policy-verified authorization to clear a raised signal, not just submitted data reasoning. "This looks legitimate" is not the same as "this is verified as authorized." The distinction matters when the failure is irreversible.

Same underlying models. Same adversarial loop. Different policy layer for the action type. The architecture did not change. The calibration did.

Why this comparison is fair

The same frontier models were used in both conditions.

We did not compare Holo against weaker baselines. We tested whether the outcome changes when the underlying models stay the same and only the decision architecture changes.

It does.

This is not a model-versus-model comparison. It is a decision-architecture comparison. The models are not the variable. The system they run in is.

The structural problem with solo judgment

Frontier models are not failing because they are bad.

They are failing because solo judgment has a structural ceiling. The fraud in the flagship case was not obvious. It was distributed across signals in a way that a single evaluation pass cannot reliably close. No single flag was sufficient. The danger existed in the relationship between fields, not in any one field alone.

This is the class of problem where solo model judgment becomes unpredictable. GPT-5.4 misses structural identity threats consistently. Claude is the most reliable solo model — but still fails on threshold cases where the attack sits in the gap between individual model heuristics. Gemini catches edge cases when stable but showed a 21% error or timeout rate under adversarial load in this study. Reliability is part of trust.

No model failed in the same place. On some scenarios, one model fails while the others catch it. On others, all three fail. Which means if you deploy a single model into a high-consequence workflow, you are deploying one specific blindspot — and you will not know which scenario exposes it until it does.

The failure modes are not random. They are structural. And they are not the same across models — which is precisely why architecture matters.

A note on intellectual honesty

This benchmark retired its own flagship result mid-study.

An earlier version of this page led with a case where all three solo frontier models failed and Holo caught it. During final validation, we reran that scenario with current model versions. All three solo models now catch it independently. The result was no longer reproducible.

We removed it.

Most benchmarks would have kept the flattering number. We did not, because a trust product that inflates its own evidence is not a trust product.

The second finding on this page — where all three solo models fail — emerged from a domain calibration refinement identified during the study. We observed a signal degradation failure mode on payment threshold cases and tightened the clearance standard for payment action types. The result was then confirmed across independent clean runs before publication. We are documenting this explicitly so the derivation is transparent.

Both results on this page are confirmed across multiple independent clean runs with current model versions. They are the results we can stand behind.

On cost

On the flagship case, Holo used fewer input tokens than either solo model that caught it.

Across the full benchmark suite, Holo uses more tokens on average. On easy cases, a trust layer does more work than necessary. That overhead is intentional — the system is not optimized for easy cases.

The efficiency advantage appears where it should: on hard cases, where faster convergence to the correct answer reduces total token spend. Wall-clock time is higher than solo models because the evaluation runs sequentially. On the flagship case, that difference is measured in seconds.

The value is not that it costs less to run. The value is that it costs less to be wrong.

Methodology

This is not a capability leaderboard.

Scenarios are constructed from publicly documented attack classes. Sources include FBI IC3, FinCEN, and CISA advisories. A result is only published if it meets all of the following conditions:

  • ✓Correct final verdict
  • ✓Correct reason for that verdict
  • ✓Appropriate severity calibration
  • ✓Clean run with no provider instability
  • ✓Stable across independent reruns

Runs affected by provider errors are excluded from public claims. We publish what survives that standard. Nothing else.
