Why this matters

Business email compromise is not a niche attack class.

It is consistently ranked by the FBI as the highest-dollar cybercrime category in the United States. It targets organizations of every size. And it is exactly the kind of decision AI systems are now being asked to support — reviewing payment requests, approving routing updates, and handling accounts payable workflows that look routine until they are not.

The cost of a wrong approval is immediate and concrete. It is a wire transfer that cannot be recalled.

If a model cannot be trusted here, it cannot be trusted at the action boundary.

The flagship result

GPT-5.4 is OpenAI's most advanced model. It is also the model most widely deployed in enterprise workflows today.

We ran it against a fraudulent payment routing update. The amount was ordinary. The language was calm. Nothing in the message labeled itself as malicious.

The threat existed only in the structural relationship between the sender identity and the approved vendor record.

A second finding

This is not just a GPT problem.

On a separate benchmark scenario, all three solo frontier models independently returned the wrong verdict. Holo returned the correct one.

That scenario tested a threshold-gaming fraud pattern: a payment request designed to stay just below an internal approval trigger while exploiting historical invoice patterns that looked individually normal but collectively suspicious.

Why this comparison is fair

The same frontier models were used in both conditions.

We did not compare Holo against weaker baselines. We tested whether the outcome changes when the underlying models stay the same and only the decision architecture changes.

It does.

This is not a model-versus-model comparison. It is a decision-architecture comparison. The models are not the variable. The system they run in is.

Every model has a blindspot

This is the central finding of the benchmark.

No single model caught everything. No single model failed in the same place.

Across multiple clean runs in this domain, the failures were model-specific and non-overlapping. What one model missed, another caught. And on some scenarios, all three solo models missed the same thing.

That matters because a company deploying a single model into a high-consequence workflow is not deploying "AI" in the abstract. It is deploying one specific blindspot and hoping it is not the one that costs them.

The benchmark is designed to make those blindspots visible before the action becomes irreversible.

The failure modes are not random. They are structural. And they are not the same across models — which is precisely why architecture matters.

A single model is not enough

The problem is not that frontier models are weak.

They are not. The problem is that their failures become unpredictable at exactly the point where mistakes become expensive.

On obvious fraud, frontier models usually get the answer right. That is not where the risk lives.

The risk lives in threshold cases — the ones that look normal at first pass, where the danger is structural rather than explicit. That is where solo judgment becomes unreliable.

Across multiple clean runs in this study, that pattern held consistently.

A single model at the action boundary is not a trust layer. It is a single point of failure with a blindspot you have not yet found.

A note on intellectual honesty

This benchmark retired one of its own strongest-looking results during validation.

An earlier version of this page led with a scenario where all three solo frontier models failed and Holo caught it. During final validation, we reran that scenario with current model versions. All three solo models now catch it independently. The result was no longer reproducible.

We removed it.

Most benchmarks would have kept the flattering number. We did not, because a trust product that inflates its own evidence is not a trust product.

The results on this page are the ones we can stand behind.

On cost

The intuitive assumption is that running multiple models costs more.

On the flagship threshold case, that assumption was wrong. Holo reached the correct verdict using fewer input tokens than either solo model that also escalated correctly.

Across the full benchmark suite, Holo uses more tokens on average. On easy cases, a trust layer does more work than a solo model needs to. That overhead is intentional.

The efficiency advantage appears where it should: on hard cases, where faster convergence to the correct answer reduces total token spend.

Wall-clock time is higher than solo models because the evaluation runs sequentially. For irreversible actions, that is not the primary risk.

The value is not that it costs less to run. The value is that it costs less to be wrong.

Methodology

Scenarios are constructed from publicly documented attack classes.

Sources include FBI IC3, FinCEN, and CISA advisories. A result is only published if it meets all of the following conditions:

  • ✓Correct final verdict
  • ✓Correct reason for that verdict
  • ✓Appropriate severity calibration
  • ✓Clean run with no provider instability
  • ✓Stable across independent reruns

Runs affected by provider errors are excluded from public claims. We publish what survives that standard. Nothing else.
